# Network Reachability Analyzer

ネットワークトポロジーをGUIで構築し、DFSベースの双方向到達性解析を実行するツールです。
トポロジーからCコードを自動生成してgccでコンパイル・実行し、全障害パターンのCSV結果とグラフを出力します。

## 概要

このツールは以下を自動で行います。

1. GUIで構築したネットワークトポロジーからCソースコードを生成
2. gccでコンパイルして実行
3. 全障害パターン（最大 2²⁸ 通り）を網羅的に検査
4. 「到達不能」となるパターンをCSVに出力
5. 到達率グラフ・SPOF・Birnbaum重要度などの信頼性解析を実施

## ツールバー操作

### Mode（編集モード）

| モード | 操作内容 |
|--------|----------|
| **Select/Move** | ノード・エッジをクリックして選択。ノードはドラッグで移動。`Delete`キーで削除、`Escape`で選択解除 |
| **Add Node** | キャンバスをクリックしてノードを追加。名前入力ダイアログが表示される |
| **Edge(A->B)** | A→B方向の有向エッジを追加。始点ノード→終点ノードの順にクリック |
| **Edge(B->A)** | B→A方向の有向エッジを追加。始点ノード→終点ノードの順にクリック |
| **Edge(Both)** | 両方向エッジを追加。A→BおよびB→Aの双方の到達性チェックで使用される |
| **Delete** | クリックしたノード・エッジを削除 |
| **Toggle Fail** | ノード・エッジの「障害対象」フラグを切り替える。実線＝対象、破線＝除外 |
| **Toggle Mandatory** | ノード・エッジの「必須」フラグを切り替える。金色リング/菱形＝必須 |

### Node Role（ノード役割）

ノード追加前に役割を選択します。

| 役割 | 色 | 説明 |
|------|----|------|
| **Source A** | 青 | A→B方向の始点ノード |
| **Source B** | 赤 | B→A方向の始点ノード |
| **Intermediate** | グレー | 中継ノード |

> 解析実行にはSource AとSource Bが各1つ以上必要です。

### ボタン

| ボタン | 機能 |
|--------|------|
| **Run Analysis** | Cコード生成→コンパイル→実行→CSV・グラフ出力・信頼性解析まで一括実行 |
| **Reliability** | 既存の `result_*.csv` に対してSPOF・Birnbaum重要度解析のみ実行 |
| **Export C** | Cソースコードをファイルに出力（コンパイル・実行はしない） |
| **Save** | 現在のトポロジーをJSONファイルに保存 |
| **Load** | JSONファイルからトポロジーを読み込む |
| **Preset** | 24ノードのサンプルトポロジーを読み込む |
| **Clear** | キャンバス上の全ノード・エッジを削除 |

## 解析の実行手順

1. **トポロジーを構築する**
   Add Node・Edgeモードでノードとエッジを配置します。

2. **役割を設定する**
   Node Roleで Source A / Source B / Intermediate を選択してノードを追加します。

3. **障害対象を調整する（任意）**
   Toggle Failモードで解析から除外したいノード・エッジを破線にします。

4. **必須経路を設定する（任意）**
   Toggle Mandatoryモードで必ず経路上に存在すべきノード・エッジを金色にします。

5. **Run Analysis をクリック**
   出力フォルダ選択ダイアログが表示されます。フォルダを選択すると自動で処理が始まります。

> **制限事項：** 障害対象コンポーネント（ノード＋エッジ）の合計は **最大28個** です。
> 28個の場合、2²⁸ ≒ 2億6800万パターンを網羅的に検査します。

## 出力ファイル

選択した出力フォルダに以下のファイルが生成されます。

### `analyzer.c`
生成されたCソースコード。トポロジー情報とDFSロジックが埋め込まれています。

### `result_N.csv`（N = 1, 2, 3, ...）
到達不能となった各障害パターンの詳細。10,000行ごとにファイルが分割されます。

| カラム | 内容 |
|--------|------|
| `failure_count` | 同時故障コンポーネント数 |
| `<ノード名/エッジ名>` | `F` = 故障、`T` = 正常 |

### `result_summary.csv`
故障数ごとの集計結果。

| カラム | 内容 |
|--------|------|
| `failure_count` | 同時故障数 |
| `unreachable_count` | 到達不能パターン数 |
| `reachable_count` | 到達可能パターン数 |
| `total_count` | 総パターン数 |
| `arrival_rate` | 到達率（0.0〜1.0） |

### `result_graph.png`（matplotlibが必要）
故障数に対する到達率（%）の折れ線グラフ。

### `result_reliability.csv`
信頼性解析の結果。コンポーネントごとのSPOF判定とBirnbaum重要度。

| カラム | 内容 |
|--------|------|
| `component` | ノード名またはエッジ名 |
| `is_spof` | `Y` = SPOF、`N` = 非SPOF |
| `unreachable_when_failed` | 故障時の到達不能パターン数 |
| `unreachable_when_working` | 正常時の到達不能パターン数 |
| `birnbaum_importance` | Birnbaum重要度 |

### `result_reliability_graph.png`（matplotlibが必要）
コンポーネントごとのBirnbaum重要度の横棒グラフ。SPOFは赤でハイライト表示。

## 解析の仕組み

### 到達性チェック

以下の **両方** を満たす場合に「到達可能」と判定します。

1. いずれかの Source A ノードから、いずれかの Source B ノードへ A→B エッジを使って到達できる
2. いずれかの Source B ノードから、いずれかの Source A ノードへ B→A エッジを使って到達できる

どちらか一方でも失敗した場合、そのパターンは「到達不能」として記録されます。

### Mandatory（必須）制約

必須フラグが設定されたノード・エッジは、有効な経路上に存在しなければなりません。
必須コンポーネントが故障している、または経路上にない場合は、他の経路が存在していても「到達不能」と判定されます。

### SPOF（Single Point of Failure）

単独で故障した場合（`failure_count = 1`）にネットワークを到達不能にするコンポーネントをSPOFと定義します。

### Birnbaum重要度

p = 0.5（各コンポーネントが等確率で故障）のときの、コンポーネントの状態が到達不能確率に与える影響度を示します。

```
Birnbaum重要度 = P(到達不能 | 故障) - P(到達不能 | 正常)
```

値が大きいほど、そのコンポーネントがネットワーク信頼性に与える影響が大きいことを意味します。

## ビジュアル凡例

### ノード

| 表示 | 意味 |
|------|------|
| 青い円（A） | Source A ノード |
| 赤い円（B） | Source B ノード |
| グレーの円 | Intermediate ノード |
| 破線の外枠 | 障害対象から除外（Toggle Fail） |
| 金色の外枠リング + "M" | Mandatoryノード（Toggle Mandatory） |
| 黄色ハイライト | 選択中のアイテム |

### エッジ

| 表示 | 意味 |
|------|------|
| 青い矢印 | A→B方向エッジ |
| 赤い矢印 | B→A方向エッジ |
| 紫の矢印 | 両方向エッジ |
| 破線 | 障害対象から除外（Toggle Fail） |
| 金色の菱形マーク | Mandatoryエッジ（Toggle Mandatory） |
